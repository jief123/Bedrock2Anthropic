通过 AWS Bedrock 调用 Claude

面向熟悉 Anthropic API 但没用过 AWS 的开发者。讲清楚 Bedrock 的裸 HTTP API、SigV4 签名、endpoint 格式，以及两套 API 的全部差异。

30 秒理解关系

Anthropic 把 Claude 模型同时部署在自己的基础设施和 AWS Bedrock 上。两套 API 调的是同一个模型，回答质量一样。区别是入口、认证、协议细节。企业选 Bedrock 通常是为了统一 AWS 账单、IAM 权限管控、数据合规。

Bedrock 的裸 HTTP API

不用 SDK 也能调。本质就是两个 POST endpoint。

Endpoint 格式

Bedrock Runtime 的 service endpoint：

Endpoint 模板
https://bedrock-runtime.{region}.amazonaws.com

示例:
https://bedrock-runtime.us-east-1.amazonaws.com
https://bedrock-runtime.eu-west-1.amazonaws.com

两个 API

用途HTTP 方法 + 路径说明
非流式 POST /model/{modelId}/invoke 同步请求，返回完整 JSON
流式 POST /model/{modelId}/invoke-with-response-stream 返回 AWS EventStream 二进制流

SigV4 认证详解

AWS 的请求签名机制。如果你用 SDK 可以跳过这节——SDK 自动处理。如果你要裸调 HTTP 或自己实现客户端，这是你需要的。

原理

每个请求都需要用你的 AWS Secret Key 对请求内容做 HMAC-SHA256 签名，放在 Authorization header 里。AWS 收到请求后用同样的算法验证签名。这比 API Key 安全——密钥本身不在网络上传输。

你需要的凭证

凭证说明来源
Access Key ID公开标识，类似用户名IAM Console 创建
Secret Access Key私密密钥，用于签名创建 Access Key 时获取，只显示一次
Session Token（可选）临时凭证才需要STS AssumeRole / SSO 获取
RegionAWS 区域如 us-east-1

签名步骤（SigV4）

SigV4 签名流程
Step 1: 构造 Canonical Request
CanonicalRequest =
  HTTPMethod + '\n' +                    POST
  CanonicalURI + '\n' +                  /model/global.anthropic.claude.../invoke
  CanonicalQueryString + '\n' +          （通常为空）
  CanonicalHeaders + '\n' +              host:bedrock-runtime.us-east-1.amazonaws.com\n
  SignedHeaders + '\n' +                 content-type;host;x-amz-date
  SHA256(RequestBody)                    body 的哈希

Step 2: 构造 String to Sign
StringToSign =
  "AWS4-HMAC-SHA256" + '\n' +
  Timestamp + '\n' +                     20260328T020000Z
  Scope + '\n' +                         20260328/us-east-1/bedrock/aws4_request
  SHA256(CanonicalRequest)

Step 3: 派生签名密钥
kDate    = HMAC-SHA256("AWS4" + SecretKey, "20260328")
kRegion  = HMAC-SHA256(kDate, "us-east-1")
kService = HMAC-SHA256(kRegion, "bedrock")
kSigning = HMAC-SHA256(kService, "aws4_request")

Step 4: 计算签名
Signature = Hex(HMAC-SHA256(kSigning, StringToSign))

Step 5: 拼 Authorization header
Authorization: AWS4-HMAC-SHA256
  Credential=AKIAIOSFODNN7EXAMPLE/20260328/us-east-1/bedrock/aws4_request,
  SignedHeaders=content-type;host;x-amz-date,
  Signature=fe5f80f77d5fa3beca038a248ff027d0445342fe2855ddc963176630326f1024
实际开发中：几乎所有语言都有 SigV4 签名库，不需要自己实现。Java 有 aws-sdk-java-v2 的 Aws4Signer，也可以只引 auth 模块做签名而不用完整 SDK。Python 有 botocore/requests-aws4auth，Go 有 aws-sdk-go-v2，Node.js 有 @aws-sdk/signature-v4。

方式一：用 AWS SDK 调用（推荐）

Java — AWS SDK for Java v2（签名自动处理）
// Maven 依赖: software.amazon.awssdk:bedrockruntime

var client = BedrockRuntimeClient.builder()
        .region(Region.US_EAST_1)
        .credentialsProvider(DefaultCredentialsProvider.create())
        .build();

String body = """
        {
            "anthropic_version": "bedrock-2023-05-31",
            "max_tokens": 1024,
            "messages": [{"role": "user", "content": "Hello"}]
        }
        """;

var request = InvokeModelRequest.builder()
        .modelId("global.anthropic.claude-sonnet-4-5-20250929-v1:0")
        .contentType("application/json")
        .accept("application/json")
        .body(SdkBytes.fromUtf8String(body))
        .build();

var response = client.invokeModel(request);
String json = response.body().asUtf8String();
System.out.println(json);

方式二：不用 SDK，只用签名库裸调 HTTP

Java — 用 aws-sdk-java-v2 的 Signer 手动签名 + HttpClient 发请求
// Maven 依赖: software.amazon.awssdk:auth, software.amazon.awssdk:regions
// 不需要引入 bedrockruntime 模块

import software.amazon.awssdk.auth.credentials.*;
import software.amazon.awssdk.auth.signer.Aws4Signer;
import software.amazon.awssdk.auth.signer.params.Aws4SignerParams;
import software.amazon.awssdk.http.*;
import software.amazon.awssdk.regions.Region;
import java.net.URI;
import java.net.http.*;

String region = "us-east-1";
String modelId = "global.anthropic.claude-sonnet-4-5-20250929-v1:0";
String url = "https://bedrock-runtime." + region
           + ".amazonaws.com/model/" + modelId + "/invoke";

String body = """
        {
            "anthropic_version": "bedrock-2023-05-31",
            "max_tokens": 1024,
            "messages": [{"role": "user", "content": "Hello"}]
        }
        """;

// 1. 构造 SdkHttpFullRequest
var sdkRequest = SdkHttpFullRequest.builder()
        .uri(URI.create(url))
        .method(SdkHttpMethod.POST)
        .putHeader("Content-Type", "application/json")
        .contentStreamProvider(() -> new ByteArrayInputStream(body.getBytes()))
        .build();

// 2. 用 Aws4Signer 签名
var signer = Aws4Signer.create();
var signerParams = Aws4SignerParams.builder()
        .awsCredentials(DefaultCredentialsProvider.create().resolveCredentials())
        .signingRegion(Region.of(region))
        .signingName("bedrock")
        .build();

var signed = signer.sign(sdkRequest, signerParams);

// 3. 用 JDK HttpClient 发送签名后的请求
var httpRequest = HttpRequest.newBuilder()
        .uri(URI.create(url))
        .POST(HttpRequest.BodyPublishers.ofString(body));

// 把签名后的 header 复制过来
signed.headers().forEach((k, vs) -> vs.forEach(v -> httpRequest.header(k, v)));

var httpResp = HttpClient.newHttpClient()
        .send(httpRequest.build(), HttpResponse.BodyHandlers.ofString());

System.out.println(httpResp.body());

Bedrock EventStream 格式

Bedrock 返回的是 AWS EventStream 二进制协议。每个帧包含 header（含 :event-type）和 payload（JSON）。解码后的 JSON payload 和上面 Anthropic SSE 的 data 字段结构完全一致。

关键点：如果你用 AWS SDK，EventStream 解码是自动的，你拿到的就是 JSON 对象。如果你裸调 HTTP，需要自己实现 EventStream 二进制解码器（协议规范）。

Java SDK 流式调用示例

Java — AWS SDK v2 流式调用（EventStream 自动解码）
// Maven 依赖: software.amazon.awssdk:bedrockruntime

var asyncClient = BedrockRuntimeAsyncClient.builder()
        .region(Region.US_EAST_1)
        .build();

var request = InvokeModelWithResponseStreamRequest.builder()
        .modelId("global.anthropic.claude-sonnet-4-5-20250929-v1:0")
        .contentType("application/json")
        .body(SdkBytes.fromUtf8String(body))
        .build();

var handler = InvokeModelWithResponseStreamResponseHandler.builder()
        .onEventStream(stream -> {
            stream.subscribe(event -> {
                event.accept(InvokeModelWithResponseStreamResponseHandler
                    .Visitor.builder()
                    .onChunk(chunk -> {
                        // chunk.bytes() 就是解码后的 JSON
                        String json = chunk.bytes().asUtf8String();
                        // json 的结构和 Anthropic SSE 的 data 字段一致
                        // 包含 type 字段: message_start, content_block_delta 等
                        System.out.println(json);
                    })
                    .build());
            });
        })
        .onComplete(() -> System.out.println("Stream complete"))
        .onError(err -> System.err.println("Error: " + err.getMessage()))
        .build();

asyncClient.invokeModelWithResponseStream(request, handler).join();

事件类型（两边完全一致）

事件说明
message_start消息开始，包含 model、usage.input_tokens
content_block_start内容块开始（text / thinking / tool_use）
content_block_delta增量内容：text_delta / thinking_delta / signature_delta / input_json_delta
content_block_stop内容块结束
message_delta消息级更新：stop_reason、usage.output_tokens
message_stop消息结束
Bedrock 额外字段：message_stop 事件中会包含 amazon-bedrock-invocationMetrics（含 inputTokenCount、outputTokenCount、invocationLatency 等）。Anthropic API 没有这个字段。
Bedrock 没有 ping 保活：Anthropic SSE API 在模型思考期间会发送 event: ping 事件（文档），防止中间链路超时断开。Bedrock EventStream 没有等效机制。做 proxy 时需要自己注入 ping 事件。

实测踩坑记录

全部通过真实 Bedrock API 调用验证

实测 · Bedrock 400

anthropic_version 值必须是 bedrock-2023-05-31

传 Anthropic 的 2023-06-01 会被拒绝。如果做 proxy，必须强制覆盖，不能保留客户端传入的值。

实测 · Bedrock 400

顶层 cache_control 会被拒绝

Anthropic API 支持请求顶层的 "cache_control": {"type": "ephemeral"}，Bedrock 不支持。block 内部的 cache_control 正常工作。

实测 · 静默降级

Server Tools 不报错但不执行

把 web_search_20250305 tool 定义发给 Bedrock，请求成功但模型不会搜索——返回"我没有搜索能力"的文本。比报错更危险。

实测 · Bedrock 400 "invalid beta flag"

Bedrock 只接受白名单内的 beta flags

Anthropic SDK 和 Claude Code 会发送 Bedrock 不认识的 beta flags（如 claude-code-20250219、redact-thinking-2026-02-12、prompt-caching-scope-2026-01-05）。原样透传会导致 Bedrock 返回 400 "invalid beta flag"。做 proxy 必须用白名单过滤，只保留 Bedrock 官方文档列出的 flags。

协议差异 · 连接超时

Bedrock 流式没有 ping 保活，长 thinking 会导致超时

Anthropic SSE API 会在模型思考期间发送 event: ping 事件保持连接活跃（官方文档）。Bedrock EventStream 没有这个机制。当模型做长时间 extended thinking 时，可能几十秒没有数据返回，中间的代理、负载均衡器或客户端会因超时断开连接。做 proxy 需要自己实现 ping 注入。

Proxy 转换清单
